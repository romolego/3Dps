# AI-маршрутизатор проекта 3Dps

## Назначение

Этот файл разводит роли между тремя сущностями:

- `ChatGPT`
- `System agent`
- `Cloud Code`/локальный dev-agent

Цель — не смешивать подготовку требований, ревизию системы и собственно разработку.

## Роли

### 1. ChatGPT

Использовать для:

- прояснения идеи;
- формулирования продуктовой логики;
- написания и улучшения ТЗ;
- структурирования запросов к разработчику;
- обсуждения вариантов до входа в код.

Выход ChatGPT:

- понятное ТЗ;
- критерии готовности;
- ограничения;
- список открытых вопросов.

### 2. System agent

Использовать для:

- аудита текущего состояния проекта;
- актуализации системных файлов и документации;
- ведения `ROADMAP.md`;
- ведения `DECISIONS.md`;
- фиксации правил и ограничений;
- сверки, что требования не противоречат текущей архитектуре.

Выход system agent:

- обновлённая системная документация;
- зафиксированные решения;
- подготовленное техническое задание для dev-agent;
- список рисков и зависимостей.

### 3. Cloud Code / dev-agent

Использовать для:

- реализации фич;
- исправления багов;
- рефакторинга;
- написания и запуска тестов;
- изменения продуктового кода.

Выход dev-agent:

- код;
- проверка работоспособности;
- перечень затронутых файлов;
- сигнал о том, какие системные документы нужно обновить.

## Правила маршрутизации

Если запрос звучит как:

- "помоги сформулировать идею/ТЗ" -> `ChatGPT`
- "зафиксируй правила, решение, ограничения, текущее состояние" -> `System agent`
- "реализуй, почини, перепиши, добавь" -> `dev-agent`

Если запрос комбинированный, рабочая цепочка такая:

1. `ChatGPT` уточняет и формализует задачу.
2. `System agent` проверяет задачу на совместимость с проектом и обновляет системный контекст.
3. `dev-agent` реализует код.
4. `System agent` при необходимости обновляет `ROADMAP.md` и `DECISIONS.md` по результату.

## Обязательный пакет передачи в разработку

Перед отдачей задачи dev-agent желательно иметь:

- цель изменения;
- пользовательский сценарий;
- ограничения;
- затронутые файлы или подсистемы;
- критерии готовности;
- решение, если оно уже принято и должно считаться обязательным.

Минимальный handoff-формат:

```text
Контекст:
Задача:
Почему это нужно:
Ограничения:
Файлы/подсистемы:
Критерии готовности:
Что нельзя ломать:
```

## Кто главный источник истины

- по смыслу продукта — `PROJECT_OVERVIEW.md`;
- по фактической архитектуре — `ARCHITECTURE.md`;
- по ограничениям — `PROJECT_RULES.md`;
- по устойчивым договорённостям — `DECISIONS.md`;
- по направлению развития — `ROADMAP.md`.

Если новая идея противоречит этим файлам, сначала нужно обновить системный контекст, а уже потом отправлять задачу в разработку.
