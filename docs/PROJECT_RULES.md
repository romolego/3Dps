# Правила и ограничения проекта 3Dps

## Продуктовые инварианты

- Проект считается `local-first`.
- Базовый режим — работа на одной машине без обязательных внешних сервисов.
- Пользователь должен понимать, где лежат его данные: основное хранилище — папка `projects/`.
- Основной сценарий проекта — подготовка и разметка кадров из видео, а не монтаж видео.
- Разделение на обычный проект и `read_only`-проект является осмысленной частью модели.

## Технические инварианты

- Канонический локальный адрес сейчас — `http://127.0.0.1:8000`.
- Канонический route проекта в браузере — `/project/<id>`.
- Запуск и остановка через `start_3dps.bat` и `stop_3dps.bat` являются частью рабочего контура.
- Структура `projects/<id>/` считается контрактом и не должна ломаться без миграции.
- Изменения форматов `project.json`, `metadata.json`, `index.json`, `markers.json`, `zones.json`, `roi.json`, `centered_index.json` требуют отдельной фиксации в `DECISIONS.md`.

## Правила для ИИ-агентов

- Сначала отделять факт из кода от гипотезы или хотелки.
- Не выдавать желаемое состояние за текущее.
- При появлении новой устойчивой договорённости заносить её в `DECISIONS.md`.
- При появлении новой идеи или приоритета заносить её в `ROADMAP.md`.
- Если меняется архитектура, обновлять документацию в том же цикле работы.
- Если задача только про ревизию системы, не смешивать её с разработкой кода без явного запроса.

## Чего не делать без отдельного решения

- не переводить проект на обязательную БД;
- не добавлять обязательную авторизацию и многопользовательский контур;
- не разносить монолит на много модулей "на всякий случай";
- не менять порт, маршруты и файловую модель молча;
- не удалять пользовательские проекты и данные автоматически;
- не принимать временные debug-вставки за норму архитектуры.

## Debug и временные вставки

По текущему аудиту в проекте есть временные диагностические механизмы:

- блоки `agent log`;
- комментарии `FIX H*`;
- запись в `.cursor/debug.log`;
- обращения к локальному endpoint `127.0.0.1:7248`.

Правило: это временный слой диагностики. Его нельзя считать обязательной частью целевой архитектуры и нельзя бесконтрольно размножать по проекту.

## Правило завершения изменения

Изменение считается системно завершённым только если:

- код работает;
- не нарушены инварианты;
- документация синхронизирована;
- при необходимости обновлены `ROADMAP.md` и `DECISIONS.md`.
